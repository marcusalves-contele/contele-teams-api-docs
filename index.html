API de Integração — Contele Equipes (Teams)

Documentação dos endpoints da API de integração do Contele Equipes. Permite gerenciar locais (POIs), portfólios, visitas (tasks), reembolsos, usuários, categorias e formulários.

Base URL https://integration.contelege.com.br/v2
Documentação oficial (Swagger) contele.github.io/api_de_integracao_contele_equipes

Legenda de status

Cada endpoint exibe um selo indicando o estado de validação:

  • ✅ Testado (200) Validado em teste real, respondeu corretamente.
  • 🔴 Bug Comportamento incorreto confirmado em teste.
  • ⏳ Não testado Endpoint de escrita (POST/PUT/DELETE) ainda não validado.

# Autenticação

Todas as requisições exigem dois cabeçalhos obrigatórios. O par precisa ser da mesma geração — um par inválido, descasado ou rotacionado retorna 403 Forbidden (confirmado em teste).

CabeçalhoValorDescrição
authorizationBearer <seu_token>Token de acesso (Bearer).
x-api-key<sua_chave>Chave de API pareada ao token.
Headers
authorization: Bearer <seu_token>
x-api-key: <sua_chave>
Atenção: token e chave precisam pertencer à mesma geração. Após rotacionar credenciais, atualize os dois valores juntos.

POIs (Locais)

Pontos de interesse — os locais cadastrados que recebem visitas.

GET /pois ✅ Testado (200)

Listar locais

Retorna a lista de locais (pontos de interesse) cadastrados.

🔒 Requer autenticação

Parâmetros de query

ParâmetroTipoObrigatórioDescrição
pageintegerSim*Página a recuperar. Divergência confirmada: o Swagger marca como opcional, mas sem ele a chamada não retorna dados.
perPageintegerSim*Itens por página. Mesma divergência — obrigatório na prática.
customIdstringNãoFiltro por ID customizado.
corporateNamestringNãoFiltro por razão social.
cnpj_cpfstringNãoFiltro por CNPJ/CPF.
idstringNãoFiltro por ID do local.
orderBystringNãoOrdenação dos resultados.
includestringNãoRelacionamentos a incluir.

* Opcional no Swagger, obrigatório na prática.

Exemplo de requisição

cURL
curl --request GET \
  --url 'https://integration.contelege.com.br/v2/pois?page=1&perPage=10' \
  --header 'authorization: Bearer <seu_token>' \
  --header 'x-api-key: <sua_chave>'

# Erros & status HTTP

Códigos de status observados nos testes e os padrões esperados em uma API REST.

CódigoSignificadoQuando ocorre
200OKRequisição bem-sucedida (confirmado nos endpoints testados).
400Bad RequestParâmetros inválidos ou ausentes (padrão REST).
401UnauthorizedCredenciais ausentes (padrão REST).
403ForbiddenPar authorization + x-api-key inválido, descasado ou rotacionado (confirmado em teste).
404Not FoundRecurso inexistente (padrão REST).
500Internal Server ErrorErro interno. Confirmado em GET /refunds/{refundId} mesmo com ID válido.

# Achados de QA (pós-testes)

Cobertura: 23 dos 24 endpoints testados — 22 funcionando (200), 1 com bug confirmado e 1 ainda pendente (POST /refunds). Inclui leitura (GET) e escrita (POST/PUT/DELETE).

✅ Funcionam (200): todos os endpoints de POIs, Portfólios, Tasks, Users e Categories; em Refunds: GET /refunds, GET /refunds/km, GET /refunds/km/{refundId} e PUT /refunds/{refundId}.
⚠️ Divergência confirmada — GET /pois: page e perPage constam opcionais no Swagger, mas são obrigatórios na prática. Recomendação: corrigir a doc oficial.
🔴 Bug confirmado — GET /refunds/{refundId}: retorna 500 Internal Server Error com refundId válido (2 IDs testados). Reportar ao time com os reference ids.
⏳ Pendente: POST /refunds (criar reembolso) — único endpoint ainda não validado.
